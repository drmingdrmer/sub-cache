# 集成测试规划

## 目标

- 验证整个分布式缓存系统在"端到端"场景下的正确性、健壮性和一致性。
- 覆盖多组件协作、并发、异常恢复、数据一致性等真实业务场景。
- 补充单元测试无法覆盖的跨模块、跨线程/进程的行为。

## 侧重点

1. **端到端数据一致性**
   - 多个 watcher/节点对同一数据区间的订阅、变更、同步。
   - 本地 cache 与远端数据源在各种变更下的一致性。

2. **并发与多客户端协作**
   - 多个 watcher 并发订阅、写入、取消、重连等。
   - 验证并发下的锁、数据竞争、死锁等问题。

3. **异常与恢复**
   - 网络中断、数据源异常、watcher crash/restart 后的恢复能力。
   - 数据源不支持订阅、连接失败、流中断等异常路径。

4. **事件顺序与幂等性**
   - 乱序、重复、丢失事件下 cache 的正确性。
   - 多次 apply 同一事件不会导致数据不一致。

5. **性能与资源释放**
   - watcher 频繁启动/停止、长时间运行下的资源泄漏检测。
   - 大量数据/高并发下的性能表现。

## 建议的集成测试场景

- **基本同步**：启动 watcher，数据源变更，验证本地 cache 自动同步。
- **多 watcher 协作**：多个 watcher 订阅同一区间，分别变更数据，验证所有 watcher cache 一致。
- **异常恢复**：watcher 运行中断（如 kill/restart），重连后 cache 能自动恢复一致。
- **不支持订阅**：数据源返回不支持订阅，watcher 能正确进入降级/等待状态。
- **并发写入**：多个客户端并发写入同一 key，验证最终一致性和 last_seq 语义。
- **事件乱序/重复**：模拟事件乱序、重复，验证 cache 幂等性和正确性。
- **资源释放**：watcher 频繁启动/停止，验证无资源泄漏（如锁、线程、内存）。

## 其他建议

- 集成测试可放在 `tests/` 目录下，使用 tokio 多任务、mock 数据源等方式模拟真实环境。
- 可用 `tokio::test` 或自定义 runtime，方便并发和异步场景测试。
- 测试应尽量自动化、可重复，避免依赖外部环境。
